# Designing an AI-driven code rewrite orchestration protocol

**The single most important insight across all the research is this: understanding old code is more valuable than writing new code.** An AI-driven rewrite orchestration system must be built around a phased protocol — comprehension first, safety net second, execution third — with deterministic workflow orchestration controlling bounded AI agent execution at every stage. The research converges on a clear architecture: a state machine governs the macro process while specialized AI agents handle micro-tasks within strict guardrails, human checkpoints, and automated quality gates. The worst mistake is starting to rewrite before you understand what the code does and have tests proving it. The second-worst mistake is attempting a big-bang rewrite instead of an incremental one.

This report synthesizes findings from Michael Feathers, Martin Fowler, Kent Beck, Neal Ford, Hyrum Wright (Google), practitioners at Airbnb, Stripe, Twitter, Shopify, and Qonto, along with academic research on AI coding agents (SWE-agent, OpenHands, MetaGPT), memory architectures, and testing methodologies. It distills these into a generalizable, language-agnostic protocol.

## Before touching a single line: the comprehension phase

The protocol begins not with coding but with systematic understanding. Michael Feathers defines legacy code as **"code without tests"** — and the systems this protocol targets have zero tests, no documentation, and tribal knowledge locked in people's heads. The comprehension phase must extract three things: structural understanding (what calls what), behavioral contracts (what the code actually does for given inputs), and implicit business logic (the rules buried in conditionals and edge cases).

**The Feathers toolkit remains foundational.** His Legacy Code Change Algorithm — identify change points, find test points, break dependencies, write tests, then make changes — provides the meta-process. His concept of *seams* (places where behavior can be altered without editing the code at that location) is critical for making legacy code testable. Object seams using interfaces are preferred; link seams and preprocessing seams serve as fallbacks. His *effect sketching* technique (drawing visual maps of what an object calls and changes) and *feature sketching* (mapping methods to the variables they use) reveal hidden responsibility groupings. Perhaps most valuable for AI-driven workflows is *scratch refactoring*: freely restructuring code to build understanding, then throwing it away — an exploratory technique that maps perfectly to AI agents that can cheaply generate and discard analysis passes.

**AI-powered comprehension compresses timelines dramatically.** Thoughtworks' CodeConcise system (documented on martinfowler.com) provides the most mature architecture: parse code into ASTs stored in a graph database, build a knowledge graph with structural and behavioral edges, then traverse the graph using LLM-powered comprehension pipelines that generate explanations at method, class, and package levels. RAG-powered Q&A over this knowledge graph enables natural-language queries like "how does authorization work when viewing card details?" Thoughtworks reports compressing legacy module reverse engineering from **six weeks to two weeks** — a 66% reduction. However, a 2025 CodeMap study found that LLMs alone cannot provide the business logic necessary to understand a codebase, and accuracy ranges from 85–95% for straightforward CRUD down to much lower for complex business logic. Financial and regulatory logic demands 100% human validation.

**Adam Tornhill's behavioral code analysis** adds a crucial dimension: mining version control history to find *hotspots* (code that is both complex and frequently changed), *temporal coupling* (files that change together, revealing hidden dependencies), and *knowledge distribution* (which developers know which parts). These techniques, implementable with the open-source Code Maat tool, identify where to focus comprehension effort — the 20% of code responsible for 80% of the risk.

For extracting implicit business rules specifically, the research identifies three complementary approaches: heuristic extraction via program slicing and conditional analysis, dynamic extraction via execution tracing and process mining, and AI-enhanced extraction where LLMs walk call chains in a knowledge graph and infer rules from patterns. The most effective LLM technique, per a 2025 study benchmarking 13 models, is **executable grounding via pseudo-code generation** — having the LLM express business logic using procedural primitives before extracting structured rules, which forces early resolution of nested conditions.

## Building the safety net before any rewrite begins

The single non-negotiable prerequisite is establishing tests that capture existing behavior. Without this safety net, every change is "edit and pray" rather than "cover and modify." For zero-test codebases, the testing strategy inverts the standard pyramid: start with broad, coarse-grained tests and progressively refine.

**Characterization tests are the primary weapon.** These capture what code actually does, not what it should do. The process is methodical: write a test with a deliberately wrong assertion, observe the failure to learn the actual output, copy that output into the assertion, rename the test to reflect your understanding, and repeat. As Feathers states, "when the official specification differs from actual observed behavior, go with actual behavior because that is what users actually depend on." For systems with too many input combinations, J.B. Rainsberger's *golden master with sampling* technique uses a seeded random number generator to create deterministic input streams, with sample size calibrated to the "fear level" — the more worried you are about getting something wrong, the larger the sample.

**Approval testing operationalizes this at scale.** Llewellyn Falco's ApprovalTests framework (available in Java, C#, Ruby, Python, JavaScript) captures complete output snapshots and uses human approval as the baseline. Emily Bache's four-part model — Arrange, Act, Print, Verify — emphasizes that the Print step (converting output to a comparable string) is crucial for useful diffs. A modern approach combining Bogus (fake data with fixed seeds) and Verify achieved **91% code coverage** with minimal effort on a legacy system. Falco's philosophy for legacy code: prioritize "CRAZY regression protection," accept that these tests are disposable bridges to proper tests, and recognize that "2 lines of code can completely lock down a legacy process."

**Property-based testing provides the strongest guarantee of behavioral equivalence.** The oracle property pattern — `assert old_implementation(x) == new_implementation(x)` for randomly generated inputs — is the most powerful technique for code migration because you don't need to know the correct answer, only that old and new agree. Frameworks like Hypothesis (Python), QuickCheck (Haskell), and jqwik (Java) generate hundreds of edge cases automatically, and *shrinking* finds the minimal failing input when a discrepancy surfaces. Academic tools like SAFEREFACTOR automate this pattern specifically for refactoring verification.

**Contract testing locks down module boundaries during incremental replacement.** Using tools like Pact, consumers define their expectations as contract files, and providers verify against them. During a Strangler Fig migration, you define contracts between the legacy facade and consumers before starting, then verify each replacement module against existing contracts. This provides decoupled, fast, deterministic verification at service boundaries — far more reliable than end-to-end tests for catching integration regressions.

**Mutation testing validates the safety net itself.** Tools like PIT (Java) and Stryker (JS/.NET) make small changes to code (replacing operators, negating conditions, removing statements) and check whether tests catch them. A high mutation score means your characterization tests actually detect behavioral changes. This is essential because line coverage lies — a test that executes code without asserting anything still counts as covered.

For production traffic capture, GoReplay records real HTTP traffic non-intrusively and replays it against test environments. Netflix used this during their REST-to-gRPC migration, uncovering memory leaks, deadlocks, and caching issues that synthetic tests missed. The workflow is straightforward: record, sanitize PII, replay against the new system, compare responses.

## Incremental rewrite patterns that keep the system running

Every authoritative source converges on one principle: **incremental migration over big-bang rewrites, always.** The research identifies three primary patterns, each suited to different scales and situations.

**The Strangler Fig pattern** operates at the system/service level. A proxy layer (reverse proxy, API gateway) sits in front of the legacy system, initially routing all traffic through. New components are built alongside the old system, and the proxy incrementally redirects specific requests to new implementations. Martin Fowler, who coined the term in 2004 after observing strangler fig vines, and colleagues at Thoughtworks define four activities: understand outcomes, break the problem into parts, deliver parts, and change the organization. An anti-corruption layer (Eric Evans' DDD concept) translates between old and new domain models. AWS recommends starting with components that have good test coverage, scalability requirements, and frequent business changes.

**Branch by Abstraction** operates at the code/module level within a single codebase. The process: create an abstraction layer capturing the interaction with the component to be replaced, update all client code to use this abstraction, build a second implementation behind it, switch clients to the new implementation (incrementally or all at once), then delete the old implementation and optionally the abstraction. Paul Hammant's critical discipline: "never add to the old style of things." ThoughtWorks' Go CI/CD tool migrated from iBatis to Hibernate over a year using this pattern, with the build pipeline failing if old patterns were detected.

**Parallel Run / Shadow Mode** provides the strongest production verification. GitHub's Scientist library is the canonical implementation: run old (control) and new (candidate) code simultaneously, compare outputs, but return only the control result to users. Custom comparators handle domain-specific equality, cleaning functions normalize before comparison, and sampling limits performance overhead. LaunchDarkly extends this with a six-phase migration flag: old → dualwrite → shadow → live → rampdown → complete, providing a controlled, reversible migration path. The critical limitation: this pattern works primarily for read operations, since running both paths for writes causes double side effects.

Kent Beck's principle — **"make the change easy, then make the easy change"** — scales micro-refactoring to system-level rewrites through *preparatory refactoring*. Before making the actual change, refactor the code into a structure that makes the change trivial. Jessica Kerr's metaphor captures it perfectly: "I want to go 100 miles east but instead of traipsing through the woods, I drive 20 miles north to the highway and then go 100 miles east at three times the speed."

The "rewrite vs. refactor" debate, framed most memorably by Joel Spolsky's 2000 essay arguing you should never rewrite, has evolved into a more nuanced consensus. **Rewrites are justified when**: the architecture is fundamentally misaligned with current needs, the technology stack is genuinely obsolete (you can't refactor Swing to React), velocity is so low that continued investment has negative ROI, or the business domain has changed so much that most old behavior is unwanted. They fail when scope is underestimated, functionality is lost, there's no rollback plan, or the big-bang approach is chosen. The strongest modern consensus: **avoid the false binary** and incrementally rewrite using the patterns above.

## The orchestration layer: how AI agents manage multi-day rewrites

The most critical architectural decision for an AI-driven rewrite system is **separating deterministic orchestration from bounded agent execution.** Agents should never decide what phase the workflow is in or what comes next. The orchestrator makes all meta-level decisions; agents execute bounded tasks within those decisions.

**The recommended architecture has two layers.** The first is a deterministic workflow engine (no LLM involvement) that enforces phase transitions, manages dependency graphs between tasks, and triggers agents at the right time. Each artifact has an explicit state machine — `draft → in-review → approved → complete` — stored as machine-readable metadata. The second layer is the agent execution layer, where specialized agents (comprehension, coding, testing, review) operate within strict boundaries. Each agent output passes through automated evaluation before the workflow proceeds. A fail loop allows 3–5 retry attempts before rolling back to human intervention. Temporal, LangGraph, and Dagster all support this pattern with durable execution, checkpointing, and event replay.

**Context management is the hardest unsolved problem in long-running AI tasks.** The research converges on hierarchical memory with explicit context engineering. Letta's key insight: "Memory is about choosing which tokens to place in your context window at any given moment." Mastra's observational memory pattern divides the context window into compressed observations (from previous sessions) and raw messages (current session), with an observer agent compressing when unobserved messages hit ~30K tokens and a reflector agent restructuring when observations reach ~40K tokens. This achieves **10x cost reduction** and outperforms RAG on long-context benchmarks. Mem0's architecture adds intelligent consolidation that merges related information and resolves conflicts, achieving 26% higher accuracy than OpenAI's memory with 90% token savings. The 12-Factor Agents guidance warns that filling a context window past **~40% causes attention fragmentation** and degraded reasoning — the "dumb zone."

For codebase-specific retrieval, **syntax-aware chunking dramatically outperforms naive approaches.** Tree-sitter-based parsing chunks code at function/class/method boundaries rather than arbitrary line breaks. The embedding model voyage-code-3 is currently recommended for highest accuracy on code. A two-step retrieval strategy — LLM-generated expanded query followed by vector search plus ColBERT re-ranking — provides the best results. Notably, Sourcegraph's Cody moved away from embeddings for very large codebases (100K+ repos) due to update complexity, finding keyword search plus LLM re-ranking more practical at extreme scale.

**Multi-agent coordination follows the supervisor pattern.** Proven agent roles include: requirements agent (breaks features into specs), architecture agent (design decisions), coding agent (implementation), testing agent (generates and runs tests), review/critic agent (validates against quality definitions), and knowledge agent (queries project context, called by others). Amazon/Anthropic research on payload referencing — tagging large code blocks with references rather than copying through the supervisor — reduced latency by **27%** and improved success rates by **23%**. For parallelization without collision, Git worktree isolation (used by OpenAI Codex and OpenHands) gives each agent an isolated copy of the codebase, with changes merged through integration branches.

**Task decomposition follows a clear hierarchy.** Continue.dev's classification is practical: Type 1 (narrow) tasks like removing feature flags need minimal context and AI handles well; Type 2 (context-dependent) tasks like debugging need the right files provided; Type 3 (open-ended) tasks like "build authentication" must be decomposed into Type 1 and 2 before assigning to AI. OpenHands' Refactor SDK operationalizes this: create an integration branch, add scaffolding and context files, assign sub-branches to agents, agents create PRs into the integration branch, human reviews and merges, final PR to main. Atlassian's 2026 case study removing feature gates across **1,400 files** validated this approach, finding that batching at the package level with tight feedback loops and well-designed AI context files minimized hallucinations.

## Human-in-the-loop modes across the autonomy spectrum

The protocol must support three interaction modes, selectable per-task based on risk and confidence.

**Fully autonomous mode** applies to low-risk, well-defined tasks (Type 1): mechanical transformations, boilerplate generation, test scaffolding. The agent executes, automated checks validate, and results queue for asynchronous review. The 12-Factor Agents principle applies: "Would I be okay if the agent did this without asking me?"

**Decision-tree mode** applies to medium-risk tasks where the agent identifies multiple valid approaches. The agent analyzes the situation, produces a structured set of options with trade-offs, and pauses for human selection. LangGraph's approval gate pattern and HumanLayer's `require_approval` decorator implement this: the agent proposes an action, execution pauses, the human approves/edits/rejects via Slack or other channels, and execution resumes. This preserves the AI's analytical power while keeping humans in control of consequential decisions.

**Pair-programming mode** applies to high-risk, complex tasks: core business logic, performance-critical paths, security-sensitive code. The interaction is synchronous with micro-questions — the agent asks specific, bounded questions as it works, similar to a junior developer working alongside a senior engineer. Confidence-based escalation automates the transition: when confidence drops below a threshold, the agent routes to a human rather than proceeding.

**The key design principle is graduated autonomy with policy-driven checkpoints.** Start with tight human oversight, progressively loosen as trust builds for specific task types. QuantumBlack's field-tested pattern has the first human touchpoint at the complete feature PR, not during individual phases — but with automated evaluation gates between every phase.

## Quality gates that enforce behavioral equivalence and design standards

Quality verification must operate at two levels: behavioral (does the new code do what the old code did?) and structural (does the new code meet design quality standards?).

**Behavioral equivalence verification** combines multiple techniques in layers. The innermost layer is unit-level characterization tests and property-based equivalence tests. The middle layer is contract tests at module boundaries. The outer layer is parallel run / shadow mode in production, using the Scientist pattern to compare old and new outputs on real traffic, with mismatch rates trending toward zero as the threshold for cutover. Traffic replay using recorded production requests provides an additional verification dimension.

**Structural quality verification uses architecture fitness functions.** Neal Ford and Rebecca Parsons' concept from *Building Evolutionary Architectures* provides the framework: objective, automated assessments of architectural characteristics integrated into CI/CD. ArchUnit (Java) and NetArchTest (.NET) encode design rules as executable tests — controllers can't access repositories directly, domain packages can't depend on framework code, no cyclic dependencies between modules. These prevent architecture erosion and serve as executable documentation of design decisions.

**Design quality metrics that actually predict maintainability** include cognitive complexity (SonarSource's metric, recommended threshold <15 per function, better than cyclomatic complexity because it accounts for nesting depth), class coupling (high coupling predicts maintenance difficulty), and code churn correlated with complexity (Adam Tornhill's hotspot analysis). Augment Code's enterprise analysis warns that "green dashboards lie" — two services reporting 90% coverage can have only 22% coverage on their shared integration paths.

**AI-powered code review against a design philosophy document is feasible but requires human oversight.** The technique is to include the philosophy document as context alongside the code diff. Research shows LLM code reviews have "moderate accuracy" and a fully automated process "may be unreliable." The recommended pattern is AI as first-pass filter with human final approval. Greptile reports that CLAUDE.md-style philosophy files have **67% adoption** among AI-assisted teams. However, up to 76% of initial LLM refactoring suggestions can be hallucinations (per GitNation analysis), making validation pipelines essential.

**Architecture Decision Records (ADRs)** document the "why" behind every design choice. Michael Nygard's format — Title, Status, Context, Decision, Consequences — creates an append-only decision log. During rewrites, ADRs are critical because decisions that seem obvious today will seem arbitrary in six months. AWS, Google Cloud, and Microsoft Azure all recommend storing ADRs as markdown files in source control, close to the relevant code.

## What real-world migrations teach about AI-driven rewrites

The case studies reveal patterns that should directly inform protocol design.

**Airbnb's Enzyme-to-RTL migration is the gold standard for AI-assisted refactoring.** They migrated ~3,500 test files in **6 weeks** versus a manual estimate of 1.5 years. The architecture was a state machine pipeline where each file moved through stages, with LLM retry loops providing contextual feedback on failure. The breakthrough insight: "There was no need to perfectly tune the initial prompt. The system learned through failure." For complex files, prompts expanded to **40,000–100,000 tokens** including nearby files, example migrations, and architectural intent. The main success driver was choosing the right related files as context, not prompt engineering.

**Qonto's Ember-to-React migration** achieved **~20x productivity gains** using a five-step CLI agent pipeline combining deterministic codemods with LLM creativity (Claude 3.5 Sonnet via AWS Bedrock). From ~50 lines of code per day manually to sometimes over 1,000 per day — roughly a 2,000% gain. The hybrid approach of deterministic mechanical fixes plus LLM-driven creative adaptation outperformed either technique alone.

**Stripe's Flow-to-TypeScript conversion** — 3.7 million lines in a single PR — demonstrates that automated codemods can handle enormous scale when the transformation is well-defined and the tooling is mature. Their team spent 1.5 years building and extending the codemod before the single-day conversion. Their dual-write database migration pattern (write to both old and new, slowly ramp up, then cut over) provides a model for data-layer rewrites.

**Google's Large-Scale Changes approach** offers the most scalable organizational model. The counterintuitive principle: **centralized infrastructure teams perform migrations, not consumers.** "If you deprecate a function, don't just tell everyone 'please upgrade.' You go into their code and make the update yourself." This eliminates unfunded mandates and leverages domain expertise. **10–20% of all code changes** at Google are the result of LSCs. Hyrum's Law — "all observable behaviors of your system will be depended on by somebody" — means any change is potentially breaking, making comprehensive testing and incremental rollout non-negotiable.

**The Netscape rewrite remains the canonical failure.** Three years without shipping, during which Microsoft completely captured the browser market. Jamie Zawinski called it "one of the biggest software disasters there has ever been." The lesson transcends technology: never stop shipping your current product during a rewrite.

## Conclusion: the complete protocol in summary

The research synthesizes into a seven-stage protocol for AI-driven code rewrite orchestration.

**Stage 1 — Comprehension**: Build a knowledge graph of the codebase using AST parsing and graph databases. Deploy AI agents to generate explanations at every level. Mine version control for hotspots, temporal coupling, and knowledge distribution. Conduct scratch refactoring to explore structure. Extract business rules using program slicing, dynamic tracing, and LLM-powered inference. Validate all AI-generated understanding with human experts, especially for complex business logic.

**Stage 2 — Safety net construction**: Establish E2E tests for critical user journeys. Write characterization/approval tests for modules to be rewritten. Set up property-based equivalence tests. Define contract tests at module boundaries. Record production traffic for replay testing. Run mutation testing to verify test quality. This stage is complete when you have sufficient confidence to detect any behavioral change.

**Stage 3 — Architecture planning**: Document target architecture and migration strategy in ADRs. Define fitness functions encoding design principles as automated tests. Choose migration pattern per component (Strangler Fig for services, Branch by Abstraction for internal modules). Decompose the rewrite into atomic, independently verifiable units. This is a human-led phase with AI providing analysis and options.

**Stage 4 — Incremental execution**: A deterministic state machine orchestrates specialized AI agents (comprehension, coding, testing, review) through each atomic unit. Each unit follows: analyze → plan → implement → test → review → merge. Git worktree isolation prevents agent collisions. Context management uses hierarchical memory with active curation (never exceeding 40% of context window). Human-in-the-loop mode varies by task risk. Retry loops with contextual feedback handle failures.

**Stage 5 — Continuous verification**: Parallel run/shadow mode compares old and new code on production traffic. Mismatch rates are tracked and must trend toward zero. Behavioral equivalence is verified through characterization tests, property-based tests, and traffic replay. Structural quality is verified through fitness functions, ArchUnit rules, and AI-assisted design review against the philosophy document.

**Stage 6 — Progressive cutover**: Feature flags control activation of new code. Canary deployments route small percentages of traffic. Monitoring tracks error rates, latency, and behavioral differences. Instant rollback is available without redeployment. Old code remains accessible until confidence thresholds are met.

**Stage 7 — Cleanup**: Remove transitional architecture (anti-corruption layers, feature flags, proxy routing). Replace characterization tests with proper unit tests. Archive golden master snapshots. Update ADRs to reflect final state. The test pyramid rights itself — many unit tests, fewer integration tests, minimal E2E smoke tests.

The three principles that should govern every decision in this protocol: **the old system is the specification** (existing behavior, including bugs, is correct until explicitly changed), **every change must be reversible** (feature flags, not deployments, control activation), and **understanding precedes action** (no rewrite step begins until the system being rewritten is comprehensively understood and tested). An AI-driven system that respects these principles can achieve the 20x productivity gains demonstrated by Airbnb and Qonto while maintaining the safety guarantees that Google's LSC infrastructure provides at scale.
